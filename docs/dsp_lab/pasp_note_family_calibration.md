# PASP Note-Family Calibration (B3–D4)

Note-family calibration extends the bidirectional PASP hammer–string model from a single note (C4) to a **local piano model** spanning neighboring notes **B3, C4, C#4, D4** with multiple velocities per note.

## Why single-note calibration is not enough

A hammer–string model can be tuned to match one reference note with physically implausible parameters. When neighboring notes require **discontinuous parameter jumps** or impossible felt/string combinations, the fit is overfitted—not a piano model.

Note-family calibration fits **one shared parameterization** whose coefficients vary **smoothly** with MIDI note number, encouraging neighboring notes to behave like one instrument.

## Parameter curves

Physical parameters are not independent per note. They are generated by **curves** evaluated at MIDI note number:

| Parameter | Typical curve type | Smoothness space |
|-----------|-------------------|------------------|
| `hammer_mass_kg` | linear | linear |
| `felt_Q0` | log_linear | log |
| `felt_p` | constant | linear |
| `string_length_m` | anchor_interpolated | linear |
| `inharmonicity_B` | anchor_interpolated | linear |
| `bridge_loss` | linear | linear |

Curve types: `constant`, `linear`, `log_linear`, `anchor_interpolated`.

Schema lives under `PASPNoteFamilyModel.params.parameterization` in graph JSON.

## Pitch authority

**MIDI note is authoritative for pitch** (`resolve_f0` uses MIDI when wired). String length, tension, and linear density modify timbre and inharmonicity; they should still produce plausible physical values but do not override MIDI pitch during family renders.

## Shared vs note-dependent parameters

- **Often shared (constant curve):** `felt_p`, `strike_position_ratio`, `soundboard_mix`
- **Note-dependent:** `inharmonicity_B`, `string_length_m`, `hammer_mass_kg`, `felt_Q0`
- **Velocity mapping (scalar on block):** `velocity_scale`, `velocity_exponent` — shared across notes in this phase

## Smoothness penalties

`compute_parameter_smoothness_penalty()` in `dsp_lab.audio.metrics.physical_plausibility` computes per-parameter:

- first-difference penalty across B3→D4 anchor values
- second-difference penalty (curvature)
- relative jump penalty

Log-space parameters (`felt_Q0`, tensions, modal losses) use log smoothness.

Calibration combines audio loss with smoothness, velocity monotonicity, and contact diagnostic penalties (configurable via `family_weights` on `CalibrationTask`).

## Reference WAVs

Place references under `data/references/piano/` (repo-relative):

```text
data/references/piano/B3_v020.wav
data/references/piano/C4_v050.wav
data/references/piano/Cs4_v080.wav
data/references/piano/D4_v100.wav
```

See `examples/calibration/pasp_family_b3_d4_reference_set.json` for the full 16-condition list (4 notes × velocities 0.2, 0.5, 0.8, 1.0).

**Do not fake calibration** when references are missing — the eval script renders and reports diagnostics only.

## Run evaluation

```bash
PYTHONPATH=src python examples/run_pasp_family_b3_d4_eval.py
PYTHONPATH=src python examples/run_pasp_family_b3_d4_eval.py --calibrate  # when references exist
```

Output: `workspace/experiments/pasp_family_b3_d4/`

- `report.md` — audio metrics, smoothness, contact diagnostics, flags
- `metrics.json` — full evaluation payload
- `parameter_curves.json` — evaluated curve values at anchor notes
- `renders/` — per-condition WAVs
- `diagnostics/` — per-condition contact JSON

## Inspect parameter curves

After eval or calibration:

```bash
cat workspace/experiments/pasp_family_b3_d4/parameter_curves.json
```

Or in Python:

```python
from dsp_lab.physics.note_family import NoteFamilyParameterSet, default_parameterization
family = NoteFamilyParameterSet(default_parameterization())
print(family.export_curve_values())
```

## Warning

A good single-note fit can be physically meaningless if neighboring notes require discontinuous or implausible parameter jumps. Note-family calibration is intended to prevent this kind of overfitting.
